9front/vz64 -- a 9front kernel port for Apple Virtualization.framework
=======================================================================

This document describes the vz64 kernel: a port of the 9front arm64
kernel that runs as a guest under Apple's Virtualization.framework
(VZ) on Apple silicon Macs.  It covers the architecture, the boot
protocol, the build process, every VZ-specific adaptation, and the
debugging lessons learned during bring-up.

Status at time of writing: BOOTS TO AN INTERACTIVE RC PROMPT,
SMP-CAPABLE.  hjfs root mounts from virtio-blk (/dev/sdF0/fs),
bootfs and factotum run, glenda logs in, init starts rc.  A
multi-cpu guest (9vz default -cpus 2, *ncpu unset) boots cleanly
with the secondary cpus online; *ncpu=1 still forces a single-cpu
boot.

    Plan 9
    timer frequency 24000000 Hz
    cpu0: Apple Virtualization.framework
    2047M memory: 822M kernel data, 1225M user, 7354M swap
    ...
    #l0: virtio10: 1000Mbps port 0x50030000 irq 65 ea dad82dd71e94
    vz.bootfs: Thu Jun 11 10:19:23 GMT 2026
    user[glenda]:
    hjfs: fs is /dev/sdF0/fs
    init: starting /bin/rc
    %

Known wrinkle: with service=terminal, termrc/profile try to bind
#i (draw) and #m (mouse), which this kernel intentionally lacks
(headless); the first rc dies on the profile and init restarts a
bare rc.  Guard the profile, or boot service=cpu and use drawterm
from the host for graphics.  Boot cmdline that works:

    console=0 nobootprompt=local!/dev/sdF0/fs

(add *ncpu=1 to force a single-cpu boot if you want to take
secondary bring-up out of the picture while debugging.)

(virtio-blk disks are lettered from 'F' by sdvirtio10's viopnp;
the first -disk is always sdF0.)


1. Overview
-----------

Apple Virtualization.framework provides a lightweight hypervisor on
Apple silicon.  Its VZLinuxBootLoader boots any kernel image that
carries the Linux arm64 Image header; the guest starts at EL1 with
a virtual GICv3, ARM generic timers, PSCI power management via HVC,
and virtio 1.0 devices on a PCI ECAM.  No emulated legacy hardware
exists at all: no 16550, no VGA, no PS/2, no SMBIOS.

vz64 is derived from the 9front arm64 (qemu-virt) kernel.  The
machine-specific layer was adapted for VZ's hardware model and a
handful of hypervisor-specific behaviors that break assumptions
baked into bare-metal ARM code.

Source layout: /sys/src/9/vz64 (conventionally bound from a host
directory during development).  Kernel configuration: "vz".

Host-side runner: 9vz, a small Go harness over the Code-Hex/vz
bindings (separate repository/directory).  It wires the virtio
console to stdio, attaches a raw disk image via virtio-blk, NAT
networking via virtio-net, an entropy device, and a vsock device,
and passes the kernel command line via the device tree.


2. Memory map
-------------

Guest RAM under VZ begins at physical (IPA) 0x70000000.  All
addresses below are physical unless noted.

    0x10000000  GICv3 distributor (GICD)
    0x10010000  GICv3 redistributors (GICR)
    0x20050000  PL031 RTC (VZ provides one)
    0x20060000  PL061 GPIO
    0x50000000+ PCI BARs (virtio devices; ECAM mapped via vmap)
    0x70000000  VDRAM: guest RAM base (up to 2GB in current config)
    0x70010000  CONFADDR: BOOTARGS area (64K)
    0x70020000  REBOOTADDR: reboot trampoline
    0x70100000  KTZERO: kernel text (load address)

Virtual layout (KZERO = 0xFFFFFFFF00000000):

    KZERO+pa            direct map of the first 4GB-ish (kaddr)
    VIRTIO = KZERO+0x10000000   device window, PA = VA-KZERO
    KMAP   = 0xFFFFFFFE00000000 high RAM mapping window
    VMAP   = 0xFFFFFFFFF0000000 vmap arena (ECAM, BARs)

The Mach structures live immediately below KTZERO (MACHADDR(n) =
KTZERO-(n+1)*MACHSIZE), and the initial page tables (L1BOT/L1/L1TOP)
below those.  See mem.h for the full picture.

mmu0init maps the entire 2GB of VDRAM with 2MB blocks up front
because VZ places the device tree blob near the top of guest RAM
and it must be readable before meminit runs.


3. Boot protocol and image format
---------------------------------

VZLinuxBootLoader requires the Linux arm64 Image header
(linux/Documentation/arch/arm64/booting.rst): a 64-byte structure
at offset 0 with executable code in the first two words and the
magic "ARM\x64" at offset 56.

vz64 embeds this header directly in l.s at _start:

    WORD $0x14000010   code0: b .+64 (branch over the header)
    WORD $0xD503201F   code1: nop
    text_offset = 0x100000  (u64)
    image_size  = 32MB      (u64)
    flags       = 0x2       (LE, 4K pages, honor text_offset)
    res2..res4  = 0
    WORD $0x644D5241   magic "ARM\x64" at offset 56
    res5        = 0

The loader places the image at RAM base + text_offset =
0x70100000 = KTZERO, so the header occupies the first 64 bytes of
kernel text and all linked addresses are self-consistent.  Entry is
at the image start; code0 branches over the header to the real
startup code.

CRITICAL SUBTLETY: code0 must be a raw WORD, not an assembler
branch (B _hdrend(SB)).  The 7l linker's follow()/xfol pass chases
unconditional branch instructions and lays the branch target's code
at the branch site, scattering the header data words elsewhere in
the text.  The linked addresses remain consistent, but the file no
longer begins with a valid header and VZ refuses to start.  Encode
the branch as data and the linker leaves everything in place.

At entry (per the Linux boot protocol): R0 = physical address of
the device tree blob, MMU state as the hypervisor left it, CPU at
EL1.  l.s saves the DTB pointer and passes it to main(uintptr dtb).

The kernel command line arrives via the DTB /chosen bootargs
property (set with 9vz -cmdline); bootargsinit parses the DTB for
bootargs, memory size, and CPU count.

Build and boot:

    # on Plan 9 (source bound at /sys/src/9/vz64)
    cd /sys/src/9/vz64 && mk
    # produces 9vz (a.out), 9vz.u (uImage via aux/aout2uimage),
    # and 9vz.bin (the uImage header stripped -- ready to boot)

    # on the mac
    ./9vz -kernel 9vz.bin -disk 9front.raw \
        -cmdline 'console=0 ...'

The $CONF.bin target strips aout2uimage's 64-byte U-Boot wrapper
(a single dd, bs=64 skip=1) as part of the build, exposing the
embedded arm64 header at offset 0.  The old manual host-side dd
step is gone.  check_kernel.sh (in the 9vz repo) validates the
header.  No further conversion is needed; the older mkarm64hdr.py
prepend-a-header approach is obsolete (and will refuse to run on a
binary that already has the magic).


4. VZ-specific adaptations
--------------------------

4.1 LSE atomics instead of LDXR/STXR

The single most important change.  Apple's hypervisor clears the
CPU's exclusive monitor on every VM exit, and VM exits happen
constantly (host timer interrupts, scheduling).  A classic
LDXR/STXR loop can therefore livelock: the STXR always fails
because the monitor was cleared between the paired instructions.
The first casualty was the kernel print lock -- the kernel hung in
quotefmtinstall() at boot, inside the very first lock() call.

tas() and cmpswap() in l.s now use ARMv8.1 LSE single-instruction
atomics, which do not involve the exclusive monitor:

    tas:     SWPW  R1, R0, [R0]   encoded as WORD $0xB8218000
    cmpswap: CASW  R1, R2, [R0]   encoded as WORD $0x88A17C02

All Apple silicon supports LSE.  The instructions are WORD-encoded
because 7a has no mnemonics for them.

4.2 No PMU

VZ guests have no performance monitor unit.  Writing PMCR_EL0,
PMCNTENSET, or PMUSERENR_EL0 traps (undefined instruction).  The
PMU setup in clockinit is removed, and lcycles() reads the virtual
counter CNTVCT_EL0 instead of PMCCNTR_EL0.  All timekeeping and
cycle counting use the ARM generic virtual timer/counter, which VZ
implements fully:

    CNTFRQ_EL0 reports 24 MHz
    CNTVCT_EL0 is the cycle/tick source (vcycles, cycles macro)
    CNTV_TVAL/CNTV_CTL drive the scheduler clock (IRQcntvns)

4.3 PSCI via HVC

Power state control is PSCI with HVC as the conduit:

    0x84000002  CPU_OFF
    0xC4000003  CPU_ON      (mpinit, secondary cpu start; note
                             the SMC64 id -- VZ rejects the SMC32
                             0x84000003 with NOT_SUPPORTED, since
                             CPU_ON takes a 64-bit entry address)
    0x84000008  SYSTEM_OFF  (vzstop debug helper)
    0x84000009  SYSTEM_RESET (exit/reboot)

hvccall() in l.s marshals a Ureg through the HVC.  The vzstop()
helper (SYSTEM_OFF + spin) deserves special mention: it terminates
the VM cleanly in a way that is observable from the host ("state:
stopped" vs. a hang), which made it the fundamental debugging
primitive during bring-up -- see section 6.

4.4 GICv3

intrinit drives the GICv3 at GICD 0x10000000 / GICR 0x10010000
(gic.c).  This works under VZ as-is.  The virtio devices deliver
PCI INTx-style interrupts through it (the ether probe above shows
irq 65).

4.5 Virtio 1.0 over PCI

All I/O is virtio 1.0 ("modern") on a PCI ECAM:

    uartvz.c        virtio-console = the system console
    ethervirtio10   virtio-net
    sdvirtio10      virtio-blk (SCSI-flavored sd)
    usbxhcipci      XHCI (if VZ exposes one)
    screen.c        virtio-gpu (with 9vz -gui; see 4.9)
    audiovz.c       virtio-sound (with 9vz -audio; see 4.10)

pciqemu.c provides ECAM config access.  Device discovery is by
vendor:device ID match (console is 0x1AF4:0x1043, gpu 0x1AF4:0x1050,
sound 0x1AF4:0x1059).

4.6 The console (uartvz.c)

VZ has no UART; the console is a virtio-console device wired by
the 9vz harness to stdio.  Bring-up is two-phase because the
console cannot exist until PCI is scanned:

    uartconsinit (early)   installs a null console; putc appends
                           to a 16K memory ring buffer
    uartvzlink (links())   probes the PCI device, negotiates
                           VERSION_1, sets up rx/tx vrings,
                           flushes the ring buffer, switches to
                           real virtio I/O, enables rx interrupts

Until links() runs, ALL print output is invisible, buffered in the
ring.  This shaped the whole early debugging experience (section 6).

TX is polled and synchronous: putc posts single characters via
descriptor 0 (iprint/panic path), kick posts batches via
descriptor 1 (queued print path).  Both wait for the device to
consume the descriptor before reusing it -- see the txwait bug in
section 6.5 for why this wait must be real.  putc (interrupt
context) and kick (process context) can run on different cpus, so
both wrap their txwait/txpost pairs in the vcon.txl ilock; this
keeps two cpus from interleaving descriptor/avail updates on the
single tx queue (required for SMP, verified clean on a 2-cpu boot).

4.9 Native graphics (screen.c)

VZ's -gui mode (in the 9vz host runner) attaches a
VZVirtioGraphicsDevice with a single scanout, a USB keyboard, and
a USB absolute-coordinate pointing device, and opens a macOS
window.  The guest needs drivers for all three.

screen.c is the virtio-gpu driver.  virtio-gpu (PCI 0x1AF4:0x1050)
is a virtio 1.0 device with two queues; we use only the control
queue (queue 0).  Bring-up, like the console, is deferred until
PCI is up: screeninit() runs from main() right after links().  It:

  * probes the device, negotiates VERSION_1, sets up the control
    queue (same vring code as uartvz.c / ethervirtio10.c);
  * RESOURCE_CREATE_2D a host resource the size of the scanout
    (format B8G8R8A8, which is XRGB32 little-endian);
  * allocates a guest linear framebuffer (the devdraw
    "softscreen") and RESOURCE_ATTACH_BACKING it -- one mementry
    per physical page, though the KZERO direct map makes kernel
    allocations contiguous so it is effectively one entry;
  * SET_SCANOUT binds the resource to scanout 0.

devdraw renders into the framebuffer in main memory.  Each
flushmemscreen(r) issues TRANSFER_TO_HOST_2D (copy the dirty
rectangle from guest backing into the host resource) followed by
RESOURCE_FLUSH (present it on the scanout).  The control queue is
polled synchronously, waiting for the device's used index before
reusing descriptors -- the same lesson as the console TX path
(section 6.5): VZ's device side is a concurrent host thread.

The cursor is software (port/swcursor.c); the virtio-gpu hardware
cursor queue is not used.  #i (draw) and #m (mouse) are in the vz
config.  Input: nusb in userspace (usbd + nusb/kb) drives the USB
keyboard and pointer; kb writes absolute events to /dev/mousein
with the 'A'/'a' prefix, which devmouse routes through
absmousetrack/scmousetrack to position the cursor.

If no virtio-gpu device is present (9vz without -gui), gpuprobe()
finds nothing, screeninit() returns quietly, and the kernel runs
headless on the serial console as before.

4.10 Audio (audiovz.c)

VZ's -audio mode (in the 9vz host runner) attaches a
VZVirtioSoundDevice with a host speaker output stream (and, with
-mic, a host microphone input stream).  The guest needs a driver.

audiovz.c is the virtio-sound driver.  virtio-sound (PCI
0x1AF4:0x1059 -- virtio device id 25, modern PCI did 0x1040+25) is a
virtio 1.0 device with four queues: controlq (0), eventq (1), txq
(2, playback), rxq (3, capture).  The driver currently implements
PLAYBACK ONLY, using controlq and txq; eventq and rxq are left
unconfigured.

It does not implement the audio(3) filesystem itself: it registers
an Audio card with the portable devaudio.c via addaudiocard(), so
the guest gets the standard #A interface (/dev/audio, /dev/volume,
/dev/audioctl, /dev/audiostat) and devaudio owns the FS and the
volume-string parsing.  This is the same split the x86 audioac97.c
driver uses; audioac97.c was the model for the userspace-facing ring
buffer (Ring / readring / writering / buffered), while screen.c and
uartvz.c were the model for the virtio side.

Bring-up, like the console and the GPU, is deferred until PCI is up:
the probe runs from audioreset (devaudio's reset, via chandevreset)
rather than from main().  It:

  * probes the device, negotiates VERSION_1 (control feature word 0
    driven to zero -- no optional sound features), sets up controlq
    and txq (same vring code as uartvz.c / screen.c), DRIVER_OK;
  * runs the PCM control lifecycle: PCM_INFO (enumerate streams,
    pick the first OUTPUT one) -> SET_PARAMS -> PREPARE.  START is
    deferred to the first period actually written, so an idle
    /dev/audio does not run a hungry stream;
  * SET_PARAMS probes a most-likely-first candidate list
    (S16_LE/48000, then /44100; stereo) the way screen.c's
    gpucreate2d() probes pixel formats -- Apple's virtio-sound
    backend is expected to be pickier than QEMU's, and this is the
    one genuine bring-up unknown (see below);
  * allocates a page-aligned userspace ring; a kproc (drainproc)
    pulls period-sized chunks (4096 B) and posts each as a txq
    descriptor chain: xfer header (stream id) -> PCM data -> status
    word, then waits on the used ring.  write() fills the ring and
    wakes the kproc; close() pads the final partial period.

Completion discipline is screen.c's: sleep on the used-ring
completion interrupt when in process context at spllo, poll during
early bring-up, never force interrupts off across a host round trip
(the throughput lesson from the GPU flush path, NOTES "GUI Bug 4").

/dev/volume is a 100%-master no-op stub: Apple's backend exposes no
guest mixer controls.

BOOT-STACK CONSTRAINT (this is why the driver is structured oddly --
read this before touching it): the devaudio reset hook (audiovzprobe,
via chandevreset) runs on the tiny per-Mach boot stack with up == nil,
BEFORE schedinit.  Bringing the virtio device live there (DRIVER_OK +
bus master) lets it raise an interrupt that nests trap frames on the
boot stack until kenter() panics; creating a kproc there is itself a
trap source.  So the probe does ONLY pure PCI discovery (sndfind) and
registers the card as PRESENT.  The ENTIRE lifecycle -- virtio
handshake AND PCM_INFO/SET_PARAMS/PREPARE AND draining -- runs in a
kproc (drainproc) created LAZILY on the first /dev/audio write
(audiovzstart, from audiovzwrite), where up != nil after schedinit on
a real KSTACK.  Plus a companion vz64 PORT fix in pciqemu.c: each PCI
slot's GIC SPI is masked until a real driver claims it (pciintrenable),
so no driverless device can interrupt on the boot stack.  See the vz64
PORT NOTES (section (e) of the working notes) for the full diagnosis.

If no virtio-sound device is present (9vz without -audio), the probe
finds nothing and #A has no card; everything else runs as before --
exactly like screen.c staying headless without -gui.

STATUS: PLAYBACK WORKS -- confirmed audible on hardware.  PCM_INFO,
SET_PARAMS and PREPARE all succeed; the lazy-kproc activation boots
clean (no kenter panic) and /dev/audio plays.  Live limitations:

  * CHOPPY on jittery network sources.  drainproc posts ONE ~4096-byte
    period (1024 frames, ~21ms at 48kHz) and BLOCKS on its completion
    before refilling, so a late ring refill (e.g. internet radio over
    the network) underruns and is heard as a gap.  The fix (anticipated
    by the "one in-flight tx period ... can pipeline later" design note)
    is to pipeline several periods on the txq and reap multiple
    completions per wakeup, and/or deepen the ring (Nperiod, currently
    8 = ~170ms).  SomaFM (stable Icecast) is noticeably smoother than
    flakier streams, but the residual gap is the driver's, not the
    source's.  NOT yet done.
  * NO /dev/audioctl OR /dev/volume CONTROLS.  The driver never sets
    adev->ctl, so any write to /dev/audioctl returns Egreg.  /dev/volume
    is a single 100%-master no-op (Apple exposes no guest mixer; volset
    ignores writes, and only "master" is in voltab).  Output is fixed at
    48000Hz/S16/2ch -- Apple also advertises S32/FLOAT and 96000Hz but
    none of that helps a 44.1kHz radio source.  Volume/quality tuning is
    a userspace concern: mixfs "mix" / zuke for volume, pcmconv -o
    s16c2r48000 to resample the source once to the native rate.
  * CAPTURE (rxq / microphone) is NOT implemented -- /dev/audio is
    write-only.  It is symmetric to playback (pre-post empty buffers on
    rxq, wait on completion, set adev->read) and is the next step; it is
    what the voice pipeline needs.  Host side also needs the mic
    entitlement + TCC consent.

Every control reply is still logged ("virtio-snd: ...", decoded by
sndstatusname) for diagnostics.

4.7 DTB parsing

bootargsinit (bootargs.c) walks the flattened device tree that VZ
passes in R0: /chosen bootargs becomes the plan9.ini-style config,
/memory reg sets *maxmem, /cpus/cpu* sets *ncpu.  VZ places the
DTB near the top of guest RAM, which is why mmu0init maps all of
VDRAM up front.

4.8 Page table coexistence (mmu.c)

l1map() skips L1 entries that already contain valid 2MB PTEBLOCK
mappings (installed by mmu0init) instead of asserting they must be
PTETABLE.  Without this, meminit -> kmapram -> l1map panicked into
the invisible early console -- a silent hang.  The mmu0init block
attributes (kernel RW, inner-shareable, write-back) are correct
for RAM, so skipping is sound.


5. What VZ provides (host harness summary)
------------------------------------------

The 9vz Go program (Code-Hex/vz bindings) configures:

    VZLinuxBootLoader        kernel + command line (DTB /chosen)
    virtio-console           on stdin/stdout, raw-ish termios
    virtio-blk               raw disk image (-disk, -ro)
    virtio-net               NAT, random locally-administered MAC
    virtio-entropy
    virtio-vsock             (for a future 9P control plane)
    virtio-gpu + USB kbd/ptr (with -gui; see 4.9)
    virtio-sound             (with -audio: host speaker output;
                              with -mic: also host mic input; see 4.10)
    -cpus N (default 2), -mem GiB (default 2)

Also supports -efi for EFI firmware boot as a fallback path.
qcow2 images must be converted to raw first.


6. Bring-up war stories (debugging notes for posterity)
-------------------------------------------------------

The early debugging environment was: no console output of any kind
before links(), no JTAG, no hypervisor trace.  The only observable
signal was vzstop() -- PSCI SYSTEM_OFF -- which makes the host
print "state: stopped" instead of hanging forever.  Boot bugs were
located by bisecting main() with vzstop probes, one VM launch at a
time.  Lessons:

6.1 The lock that wasn't (LDXR/STXR livelock)

First hang: quotefmtinstall(), the first lock() in the boot path.
The exclusive-monitor livelock (4.1) means ANY spinlock can hang a
VZ guest even with zero contention.  If a bare-metal ARM kernel
hangs mysteriously early under a hypervisor, suspect the monitor.

6.2 The 64-byte lie (image load offset)

For a long stretch, static variables appeared to contain garbage
at boot: a Lock in the data segment was nonzero, zeroing it by
address fixed one hang only to reveal another.  Heroic theories
about cache writeback, MMU-disable being trapped, and linker BSS
placement were tested and demolished one by one (skipping every
physical-address operation in l.s changed nothing; zeroing all of
BSS from C changed nothing).

The actual bug: the image was being loaded 64 bytes off from the
linked addresses by an external header-prepending tool with a
wrong text_offset.  Every memory reference was self-consistent
within the running kernel but shifted relative to the loaded
image, so initialized data appeared corrupt.  One header fix and
every "memory corruption" symptom evaporated.

Moral: when symptoms say "memory is garbage but only sometimes",
check the load address against the link address before anything
else.

6.3 The linker ate my header (7l follow pass)

Embedding the arm64 Image header with "B _hdrend(SB)" as code0
produced a binary whose header words were scattered: 7l's branch-
following optimization relocated code into the header area.  Raw
WORD encoding for code0 is mandatory.  (Detected because
check_kernel.sh found garbage at the magic offset.)

6.4 Small statics live in the data segment

Plan 9 linkers promote uninitialized globals smaller than MINSIZ
(64 bytes) into the data segment (dodata pass 1).  A static Lock
is NOT between edata and end, and zeroing edata..end does not
touch it.  This mattered during the load-offset hunt and is worth
knowing whenever "BSS" assumptions are in play on Plan 9.

6.5 txwait that didn't wait

The virtio console's txwait() was a no-op (it advanced its own
bookkeeping counter instead of waiting for the device's used
index).  putc/kick then reused the descriptor and buffer while
the asynchronous VZ backend was still reading them.  Symptoms:
console text duplicated in blocks, then with individual characters
doubled/interleaved ("110000MMbbpss..."), then TX wedging entirely
-- which looked exactly like a kernel hang.  The fix is to spin
until used->idx == avail->idx before reusing anything.

Moral: under VZ the device side is a host thread, genuinely
concurrent with the guest.  Polling drivers ported from
synchronous-feeling environments (QEMU TCG) must actually wait.

6.6 PMU is a trap (literally)

See 4.2.  Writing PMU system registers undef-traps under VZ.  The
writes were vestigial in clockinit but fatal once vectors were up,
panicking into the invisible pre-links console buffer.

6.8 SMP: the redistributor that wouldn't be found

Bringing up cpu1 hit a chain of failures, each masking the next.
PSCI CPU_ON first returned -1 (NOT_SUPPORTED): the SMC32 id
0x84000003 is wrong for a call that takes a 64-bit entry address;
VZ wants the SMC64 id 0xC4000003 (what Linux uses).  The
argument-less PSCI calls (SYSTEM_OFF/SYSTEM_RESET) only exist as
SMC32 and had worked all along, which is why the conduit looked
fine until CPU_ON.

With CPU_ON fixed, cpu1 started and ran to intrinit -> getrregs,
where it panicked: "no re-distributor for cpu 1".  getrregs
identified a cpu's GICv3 redistributor by matching GICR_TYPER
against machno.  Several theories about the match field
(Processor_Number vs affinity) and the frame stride (0x20000 vs
the GICv4/VLPI 0x40000) were tried and failed, each producing the
same panic.

Diagnostic tracing (a dumprregs() that walks and prints every
frame) settled it.  The dump showed BOTH cpu0's and cpu1's frames
reporting GICR_TYPER ALL-ZERO -- affinity word, Processor_Number,
Last and VLPIS all read 0.  Nothing in the GIC distinguishes cpu1
from cpu0; affinity matching is simply impossible under VZ.  cpu0
"worked" only because want==0 accidentally equalled the zeroed
field.  But the frames ARE laid out one-per-cpu, in cpu order,
0x20000 apart, with the region terminated by unmapped all-ones
space (a 0xffffffff read that decodes as Last+VLPIS -- the red
herring that had fed the stride theories).

The fix stops reading identity out of a zeroed TYPER and uses the
one fact the dump proved: positional layout.  getrregs prefers a
real affinity match (for actual hardware / qemu) but only counts
it when the field is non-zero, then falls back to indexing the Nth
frame by machno, bounded by the DTB-advertised GICR region.  Two
companion fixes were needed: GICR_WAKER.ProcessorSleep had to be
cleared on the secondary before its SGI/PPI frame would respond
(else no timer interrupts, silent wedge), and bootargs.c had to
identify the GIC node by its `compatible` = "gic-v3" string rather
than node name so gicrbase/gicrsize actually populated.

Moral: when a hardware identity register reads all-zero, stop
trying to match on it.  Trust the structural layout the hardware
actually exposes, and add a dumper before guessing a fourth time.

6.9 General technique

  - vzstop() bisection: move a single probe through main(),
    rebuild, relaunch.  Clean stop = code above the probe works.
  - One change per round.  Two theories tested at once cost more
    than they save when both can hang identically.
  - Distrust "it hangs" -- with a dead console TX queue, a healthy
    kernel looks identical to a wedged one.  After the console is
    live, "hangs" may just be output stalls.
  - The build/boot loop spans two machines (Plan 9 build, mac
    host).  Stale-binary mistakes (forgetting dd, the python tool
    silently refusing and leaving an old .img) produced identical
    output across "different" tests twice during bring-up.  When
    output doesn't change at all, suspect the artifact pipeline,
    not the code.


7. Resolved milestones and remaining TODO
-----------------------------------------

7.0 Resolved (was "Known issues" in earlier drafts)

  - Userspace bring-up: DONE.  chandevreset, virtio-blk
    discovery, userinit, schedinit, init0 and the boot process
    all work; the kernel reaches an interactive rc prompt on an
    hjfs root over virtio-blk.
  - SMP: DONE and verified.  A 2-cpu guest (9vz default -cpus 2,
    *ncpu unset) boots cleanly to the rc prompt with cpu1 online.
    The decisive fixes:
      * PSCI CPU_ON needs the SMC64 id 0xC4000003 (the SMC32
        0x84000003 returns -1 NOT_SUPPORTED under VZ).
      * getrregs cannot match a redistributor by affinity under
        VZ -- GICR_TYPER reads ALL-ZERO for every frame.  It now
        uses a POSITIONAL fallback: cpu N takes the Nth 0x20000
        redistributor frame, bounded by the DTB-advertised GICR
        region.  (See section 6.8 for the full diagnosis.)
      * GICR_WAKER.ProcessorSleep must be cleared on a secondary
        (and ChildrenAsleep observed low) before its SGI/PPI
        frame responds, or the secondary never gets a clock.
      * bootargs.c identifies the GIC node by its `compatible`
        property containing "gic-v3" (not by node name), so the
        gicrbase/gicrsize that bound the walk actually populate.
      * main.c BSS clear gated to cpu0; uartvz TX serialised with
        the vcon.txl ilock (section 4.6).
    CPU count is NOT limited to two: conf.nmach comes from the
    DTB / *ncpu, the redistributor walk and cpumpid[] are sized
    by MAXMACH, and getrregs indexes by machno.  To try more
    cpus, launch 9vz with -cpus N (N <= the framework's
    maximumAllowedCPUCount, typically the host physical-core
    count) and leave *ncpu unset.  Bring-up instrumentation is
    retained but gated behind *gicdebug / *smpdebug (quiet by
    default); see NOTES for what each switch prints.
  - The out-of-tree print.c workaround (fmtlclear/fmtlwhere) has
    been REVERTED to stock: the helpers are gone from
    /sys/src/9/port/print.c, the declarations are gone from
    fns.h, and the fmtlclear() call is gone from main.c.  Boot
    without it works (the load-offset fix was the real cause of
    the "fmtl garbage").  There is no longer any modification to
    shared port code.
  - uartvz TX vring race between putc and kick: FIXED with the
    vcon.txl ilock around txwait/txpost in both paths.  Verified
    clean on a 2-cpu boot (no garbled console).

7.1 Remaining TODO

  - The C-level BSS zeroing loop at the top of main() (gated to
    cpu0) may be redundant now that the load offset is fixed and
    l.s clears BSS; harmless to keep, untested whether removable.
  - The early-boot ring buffer flush order vs. rx interrupt
    enable could drop console input arriving during flush
    (cosmetic).
  - rebootcode/reboot path untested under VZ (PSCI SYSTEM_RESET
    works for exit; warm reboot into a new kernel is unexplored).
  - Native graphics: DONE.  screen.c is a virtio-gpu driver that
    implements the devdraw screen.c contract; #i/#m/swcursor are
    in the vz config; USB keyboard/pointer are handled by nusb in
    userspace.  Launch 9vz with -gui.  See section 4.9 below.
    Remaining: hardware cursor (uses swcursor today).  Mode set:
    screeninit now issues GET_DISPLAY_INFO as its first control-
    queue round trip and, absent an explicit vgasize, adopts
    scanout 0's host-preferred size; vgasize=WxHx32 still
    overrides.  The service=cpu + drawterm-from-host path still
    works and needs no display device.

    BRING-UP NOTE (first -gui boot): the initial attempt printed
    only "virtio-gpu: scanout setup failed" and then init died on
    the downstream binds -- "#i: no frame buffer" and "#m: ... the
    front fell off" (Egreg from mouseattach).  Both downstream
    errors are CONSEQUENCES of the scanout failure: conf.monitor
    is set 1 only after setup succeeds, so without it #i has no
    gscreen and #m's attach errors.  Fixing the virtio-gpu bring-
    up fixes all three.  screen.c now prints, per failing step,
    the exact virtio-gpu response code (create-2d / attach-backing
    / set-scanout / get-display-info, decoded by gpurespname), the
    matched PCI device (%T did/rid), the scanout geometry, and
    checks the FEATURES_OK handshake -- so the next boot localises
    the failure instead of just saying "failed".
  - Audio: PLAYBACK WORKS (confirmed audible on hardware).  audiovz.c
    presents the audio(3) #A interface via devaudio.c; the host wiring
    exists (9vz -audio / -mic).  PCM_INFO/SET_PARAMS/PREPARE succeed and
    the lazy-kproc activation boots clean.  Remaining work, in order of
    leverage:
      * txq PIPELINING -- the live limitation.  drainproc posts ONE
        ~21ms period and blocks on completion before refilling, so a
        late ring refill (jittery network radio) underruns and the
        audio is choppy.  Pipeline several periods on the txq + deepen
        the ring (Nperiod).  This is the one fix that addresses the
        choppiness at the root rather than working around it.
      * CAPTURE (rxq / microphone) -- not implemented; /dev/audio is
        write-only.  Symmetric to playback; what the voice pipeline
        needs.  Host side also needs the mic entitlement + TCC consent.
      * (optional) wire adev->ctl / a real voltab if guest-side
        speed/volume control is ever wanted -- today /dev/audioctl
        returns Egreg and /dev/volume is a no-op (Apple exposes no
        mixer); volume/quality is done in userspace (mixfs/zuke/pcmconv).
    See section 4.10 and /usr/dave/9vz-audio-and-fullscreen.md
    section (c).
  - Higher cpu counts (>2) are plausible but unverified -- the
    code has no 2-cpu assumption, but only -cpus 2 has been
    exercised.  Use *gicdebug/*smpdebug if a higher count
    misbehaves.
  - The svcmode EL2 path in l.s is dead code under VZ (guests
    enter at EL1) but retained for portability.
